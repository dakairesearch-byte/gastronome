# the-chef
**Lens:** Restaurant owner — is JoJo represented fairly, are dishes attributed correctly, would I trust this app to send customers? **Reviewed:** `restaurant-desktop.png`, `review-new-desktop.png`, `community-desktop.png`, `src/app/restaurants/[id]/page.tsx`, `src/app/review/new/page.tsx`.

## Top 5 findings

1. **[P0] What's wrong:** "What Reviewers Mention" section is empty for JoJo — the page has a Ratings Dashboard and "On Social" but no signature-dish chips, no menu, no price range, no hours (`restaurant-desktop.png` mid-page is blank between Ratings and On Social). **Why it matters:** A 1-Michelin-star Jean-Georges restaurant with zero visible dish info reads as "we don't actually know this place." A customer sets expectations from this page and arrives unprepared — bad for both of us. **What to do:** When `dishes.length === 0` (`page.tsx:501`), show a graceful state: "Menu coming soon" plus a "View menu on restaurant site" link using `restaurant.website` (already loaded, `page.tsx:388`). **Why you'd want to do this:** Empty middle of a flagship page is the single most embarrassing thing on the screen. (effort: S)

2. **[P0] What's wrong:** A raw Google Cloud error message ("Google Maps Platform rejected your request. This API is not activated...") is rendered as page content in the sidebar (`restaurant-desktop.png` top-right). **Why it matters:** Customers think *my restaurant* is broken, not your integration. They bounce, or worse, share the screenshot. The map iframe (`page.tsx:807`) has no error boundary — a billing/key failure leaks the provider's stack to my customers. **What to do:** Wrap iframe in a try-render fallback; on any error show a neutral "Map unavailable — open in Google Maps" link using `restaurant.google_url` (already loaded). **Why you'd want to do this:** Never let an infra error appear above the fold on a paid-tier restaurant page. (effort: S)

3. **[P1] What's wrong:** Reviews collapse into a single "avgRating" number that mixes any new user-submitted star with the existing aggregated `google_rating` (`review/new/page.tsx:236-247`). A drive-by 1-star wipes weeks of accumulated signal because the divisor is `review_count + 1`, not the source-count base. **Why it matters:** One angry customer can tank my displayed rating overnight, and I have no way to flag, respond, or contest. That's the difference between trusting this aggregator and refusing to be listed. **What to do:** Store user reviews in a separate field/aggregate (Gastronome rating), keep Google/Yelp/Infatuation untouched in the dashboard, add a "Restaurant response" affordance on each review. **Why you'd want to do this:** Recoverability is what makes Yelp tolerable for owners. Without it, ambitious restaurants will ask to delist. (effort: M)

4. **[P1] What's wrong:** "The Story" is the restaurant's only narrative surface, and it's not editable by the owner — comes from `restaurant.description` (`page.tsx:761-773`), source unknown. JoJo's description ("Elegant townhouse restaurant by Jean-Georges Vongerichten with a seasonal French menu") is generic and undated. **Why it matters:** Menus change seasonally (the copy even *claims* "seasonal"); chef turnover happens; the story is the only place to convey what's true today. **What to do:** Add an owner-claim flow ("Claim this restaurant") that unlocks `description`, hours, current chef, and a `last_updated_at` timestamp shown on the page. **Why you'd want to do this:** Owner-curated copy is more accurate and creates a relationship instead of an extractive one. (effort: L)

5. **[P2] What's wrong:** Review form (`review-new-desktop.png` / `review/new/page.tsx:438-456`) lets any signed-in user post a Quick Review with only a star — no dish field, no visit date, no party size. **Why it matters:** "3 stars, brunch was meh" with no dish or date is unactionable for me and unhelpful to a future diner deciding tonight. Also: a user can create a brand-new restaurant entry (`page.tsx:165-189`) with zero verification — duplicate JoJos, wrong addresses. **What to do:** Add optional "dish ordered" (autocomplete against `restaurant_menu_items`) and "visit date" fields; gate new-restaurant creation behind Google Places match. **Why you'd want to do this:** Specific reviews are recoverable; vague ones aren't. (effort: M)

## Quick wins (max 5, no severity)
- Add price range ($–$$$$) and a hours-today line to the hero (`page.tsx:327-374`) — basic info every diner checks before booking.
- Show "Last updated" on `Ratings Dashboard` (`page.tsx:447`) so I can tell if Google 4.3 is from today or last quarter.
- Make the Instagram chip (`page.tsx:400-418`) link to the *restaurant's* IG, not just the handle — confirm `instagram_url` isn't null before showing.
- Add a "Reservations" button to the hero if the website resolves to OpenTable/Resy — pull-through traffic is the only reason I'd promote this page.
- Add a `noindex` fallback or skeleton when `scoreSources.length === 0` so a half-enriched restaurant doesn't get crawled in an embarrassing state.

## Bigger bets (max 2)
- **Restaurant-claim + owner dashboard.** Let verified owners edit description, hours, menu pointer, photo selection, and post one pinned "Note from the kitchen" per month. **The tradeoff:** It's product-and-trust-and-spam work (verification, moderation, abuse). Without it, owners view this app as scraping them; with it, they promote you.
- **Dish-level reviews tied to menu items.** Require a dish tag on every star rating, link to `restaurant_menu_items`, surface "dish satisfaction" per item on the page. **The tradeoff:** Higher friction on review submission may reduce volume short-term, but converts "Reviewers Mention" from a chip cloud into actually-trustworthy dish signals.

## Alarming
JoJo's page has Michelin-star prominence but zero menu, zero hours, zero price, and a visible Google error — a Michelin chef would not want a customer's first impression to be this.
